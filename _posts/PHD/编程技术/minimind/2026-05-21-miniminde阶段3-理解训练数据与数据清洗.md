MiniMind 这一阶段非常重要，因为模型结构只决定“容量和计算方式”，而数据决定模型到底学到什么。

MiniMind 官方项目明确包含 tokenizer、数据清洗、预训练、SFT、LoRA、DPO、RLAIF、Tool Use 等完整链路，并提供经过收集、蒸馏、清洗、去重后的全阶段数据集。 

MiniMind 数据集页面也说明，MiniMind-3 主线预训练数据已经整理为统一的 text -> next token prediction 格式，目标是在小算力下兼顾文本质量、长度分布、中英混合能力，以及与后续 SFT / Tool Calling / RLAIF 的模板衔接。



# 第3阶段总目标

```
1. 看懂 MiniMind 的 tokenizer / pretrain / SFT / DPO / RLAIF 数据格式；
2. 能解释为什么同样模型结构，不同数据会训练出完全不同的模型；
3. 能写出自己的数据清洗 pipeline；
4. 能构造一个小规模高质量领域数据集；
5. 能通过训练 loss、token 分布、重复率、样本长度、生成结果来判断数据质量；
6. 能为后续 LoRA / SFT / DPO 实验准备数据。
```



数据集的理解

```
原始文本
→ 过滤
→ 去重
→ 长度控制
→ 模板统一
→ tokenizer 编码
→ dataloader
→ loss mask
→ 训练效果
```

构建 loss mask 示例：

让**模型只在“助手回答”的部分计算损失，忽略“用户提问”的部分。**

```python
from transformers import AutoTokenizer
tokenizer = AutoTokenizer.from_pretrained("../model/")#从model下的tokenizer.json文件中加载tokenizer

input_text = "<user>请解释 RoPE</user><assistant>RoPE 是旋转位置编码...</assistant>"
input_ids = tokenizer.encode(input_text)
labels = [-100]*len(input_ids)  # 默认全 -100
assistant_start = input_text.find("<assistant>")
assistant_end = input_text.find("</assistant>")+len("</assistant>")
assistant_ids = tokenizer.encode(input_text[assistant_start:assistant_end])
labels[assistant_start:assistant_end] = assistant_ids
print("input_ids:", input_ids)
print("labels:", labels)
```

```
input_ids: [63, 832, 311, 65, 960, 2093, 1243, 114, 83, 72, 4573, 832, 311, 65, 63, 1388, 570, 811, 65, 85, 114, 83, 72, 4879, 4318, 1192, 2303, 5338, 4548, 4573, 1388, 570, 811, 65]
labels: [-100, -100, -100, -100, -100, -100, -100, -100, -100, -100, -100, -100, -100, -100, -100, -100, -100, -100, -100, -100, -100, 63, 1388, 570, 811, 65, 85, 114, 83, 72, 4879, 4318, 1192, 2303, 5338, 4548, 4573, 1388, 570, 811, 65]
```

我们可以观察到，labels是将user的信息做了个屏蔽，全替换为-100了，保留了assistant的信息

在 SFT 时，我们把一整段对话 `[用户提问 + 助手回答]` 拼在一起喂给模型。

- **如果不加 mask**：模型会把“用户提问”也当作需要预测的目标。这会浪费大量计算力去学习如何生成用户问题，而且会破坏模型学到的对话规律（模型会以为自己应该接着生成用户的话，而不是回答）。
- **加上 mask**：我们只关心模型能不能在看到 `<user>问题</user>` 后，正确预测出 `<assistant>答案</assistant>`。所以要把提问部分的 loss 屏蔽掉。

# 单元 4：理解数据清洗

**目标：**明确噪声、重复、格式错误对训练影响。

可过滤项

```
- 空文本
- 重复
- HTML/URL
- 乱码
- 低信息密度
- 异常长度
```

- `minimind-3` 主线推荐采用 `pretrain_t2t` + `sft_t2t` + `rlaif/agent_rl` 的阶段式训练组合。
- 想要最快速度从0实现Zero模型，推荐使用`pretrain_t2t_mini.jsonl` + `sft_t2t_mini.jsonl` 的数据组合
- 推荐具备一定算力资源或更在意效果的朋友完整复现 `minimind-3`；仅有单卡GPU或更在意快速复现的朋友强烈推荐 mini 组合。
- 当前 `sft_t2t / sft_t2t_mini` 已经混入 Tool Call 数据，因此通常不需要再额外做一轮独立的 Tool Calling 监督微调。

```
数据集A：原始
数据集B：故意加入重复
数据集C：清洗后
输出预期：重复数据 loss 更低，但生成质量下降
```

```
清洗目的：
- 保证数据质量，提高训练效果
- 去除噪声和无效文本
- 控制样本长度和格式一致性

噪声类型：
- HTML、JS、CSS 标签
- URL、网页导航、广告
- 重复文本
- 乱码或低信息密度文本
- 异常长度（过短或过长）

去重策略：
- Exact match 去重
- 后续可用 simhash 或 n-gram 近似去重

潜在误杀风险：
- 误杀高符号密度的代码或公式
- 误杀短文本但有高信息量的样本
- 模板标记或特殊 token 被误删

思考与结论：
- 清洗策略应平衡“噪声去除”和“保留信息”
- 去重可以减少模型背诵重复，但训练 loss 可能升高
- 对不同阶段的数据（Pretrain/SFT/DPO）应使用不同规则
```





# 单元 5：实现数据分析器

**目标：**写脚本分析 jsonl 数据质量。

```python
import json, hashlib, re
from collections import Counter

def md5(text):
    return hashlib.md5(text.strip().encode()).hexdigest()

def analyze(path, max_lines=10000):
    lengths, duplicates, empty= [],0,0
    hashes = Counter()
    with open(path,"r",encoding="utf-8") as f:
        for i,line in enumerate(f):
            if i>=max_lines: break
            obj = json.loads(line)
            text = obj.get("text","")
            lengths.append(len(text))
            if not text: empty+=1
            hashes[md5(text)]+=1
    duplicates = sum(c-1 for c in hashes.values() if c>1)
    print(f"样本数: {len(lengths)}, empty: {empty}, duplicates: {duplicates}")
    print("avg length:", sum(lengths)/len(lengths))

analyze("./pretrain_t2t_mini.jsonl")
```

这个比较捞，就记录重复和是否为空。



```

分析器路径：
- `scripts/analyze_dataset.py`

样本数量：
- 输入 jsonl 文件总条数，例如 100,000 条

空文本比例：
- 统计空字符串的条数 / 总条数
- 示例：0.5%

重复比例：
- Exact duplicate 数量 / 总条数
- 示例：2%

长度分布：
- 平均长度：512
- p50/p90/p95/p99：400 / 800 / 1200 / 2000

结论：
- 数据大部分符合长度要求
- 重复样本少，空文本比例低
- 可以继续做进一步清洗
```



# 单元 6：实现清洗 pipeline

**目标：**把规则落地到脚本。

```python
def clean_pretrain(text):
    text = text.strip()
    if len(text)<50: return None
    if len(text)>8000: text=text[:8000]
    if re.search(r"<script|http|https|</html>",text,re.I): return None
    return text

with open("./pretrain_t2t_mini.jsonl","r",encoding="utf-8") as f, open("./pretrain_cleaned.jsonl","w", encoding="utf-8") as out:
    for line in f:
        obj = json.loads(line)
        cleaned = clean_pretrain(obj.get("text",""))#清洗文本，移除空格、HTML标签等
        if cleaned: out.write(json.dumps({"text":cleaned},ensure_ascii=False)+"\n") #重新写回去
```

```
清洗规则：
- JSON 合法性
- 字段完整性（text / messages / role）
- 字符长度过滤：min_chars=50, max_chars=8000
- 噪声过滤：HTML 标签、URL、过多符号
- Exact duplicate 去重

过滤阈值：
- Pretrain: 50–8000 字
- SFT: user >=2, assistant >=10, max 6000 字
- symbol_ratio <= 0.35

被过滤比例：
- 总样本中删除比例，例如 8%
- 不同类型的过滤统计

示例（保留/删除）：
- 保留：高质量文本或指令-回答对
- 删除：含 <script> 标签或大量乱码的文本

改进意见：
- 可针对代码 / 数学公式样本调整 symbol_ratio
- 对 SFT 模板添加 role 检查，避免 loss mask 错误
- 可后续加入 simhash 去重
```

# 单元 9：数据混合比例实验

**目标：**理解不同数据混合比例的影响。

```

混合比例：
- 90% 通用 + 10% 领域
- 70% 通用 + 30% 领域
- 50% 通用 + 50% 领域

用小模型训练或生成测试：
- 检查领域能力
- 检查通用能力


样本量：
- 预训练：50MB–200MB
- SFT：5k–20k 条

领域能力观察：
- 随比例增加，领域问题回答准确率提升
- 术语覆盖率提升

输出观察：
- 领域比例过高，小模型容易偏向专业术语，生成回答在通用问题上下降

结论：
- 领域数据比例不能过高
- 小模型更容易被高比例领域数据偏移
- 最佳比例需平衡领域能力与通用能力
```
